# Doing KiCad schematics & layout as Claude Code

> For a future Claude session. If you concluded "I can't build the schematic/PCB
> because KiCad has no scripting API" — that conclusion is wrong. This explains
> what to do instead. This repo already captured a full 3-channel schematic this
> way (ERC 0/0); see [hardware/gen_ref200_rtd_sch.py](../hardware/gen_ref200_rtd_sch.py)
> and the [Track E log](sessions/trackE.md).

## The mental model (read this first)

You do **not** drive the KiCad GUI and you do **not** need a live scripting API.

The thing people mean by "no API" is real but narrow: **Eeschema (the schematic
editor) exposes no Python API.** That is the *only* true part. It does not matter,
because:

1. **Every KiCad file is plain-text S-expression** that you can read and write
   with ordinary file tools (Read / Write / Edit), or generate with a Python
   script. `.kicad_sch`, `.kicad_pcb`, `.kicad_sym`, `.kicad_mod`, `.kicad_pro`
   (JSON) — all text. Open [hardware/ref200-rtd.kicad_sch](../hardware/ref200-rtd.kicad_sch)
   and look: it is just `(symbol ...)`, `(label ...)`, `(wire ...)` nodes.
2. **`kicad-cli` is a headless validator and exporter.** It runs ERC, DRC,
   netlist export, BOM, SVG/PDF render, gerbers, STEP, etc. from the command
   line with no GUI. This is your feedback loop — it replaces "look at the
   editor" with "read the report."

So the workflow is: **author the file as text → validate with `kicad-cli` →
read the report → fix the text → repeat.** That is a normal code/test loop, and
it is the same loop you already use for everything else.

## The toolchain on this machine

- `kicad-cli` is **KiCad 10.0.3**, installed but **not on PATH**:
  `C:/Program Files/KiCad/10.0/bin/kicad-cli.exe`.
- The repo's gate scripts locate it for you. Just run them:
  ```bash
  bash scripts/run_gates s005      # ERC + DRC, JSON reports under reports/, exit nonzero on errors
  bash scripts/erc.sh              # ERC only      -> reports/erc/erc_<tag>.json
  bash scripts/drc.sh              # DRC only      -> reports/drc/drc_<tag>.json
  bash scripts/netlist.sh          # export netlist -> sim/netlists/ref200-rtd.net
  ```
  Locator + version-pin logic lives in [scripts/lib/common.sh](../scripts/lib/common.sh);
  it falls back to the known install dir, so you don't hard-code the path.
- Subcommands worth knowing: `sch erc`, `sch export {netlist,bom,svg,pdf}`,
  `pcb drc`, `pcb export {gerbers,drill,pos,step,svg,pdf}`, `pcb render`.

## Schematic — how it was actually done here

The schematic was generated by a Python script that **writes the `.kicad_sch`
S-expression directly**: [hardware/gen_ref200_rtd_sch.py](../hardware/gen_ref200_rtd_sch.py).
The techniques that make this robust:

1. **Reuse real symbol bodies.** The script reads the project library
   [libraries/symbols/ref200-rtd.kicad_sym](../libraries/symbols/ref200-rtd.kicad_sym)
   and KiCad's stock `power.kicad_sym`, and copies the exact `(symbol ...)`
   blocks into the schematic's `(lib_symbols ...)` section (see `extract_symbol`).
   You do not hand-draw symbol geometry.

2. **Place each component at rotation 0**, then **connect pins by dropping a net
   label / power symbol / no-connect flag at the *exact* pin coordinate** —
   instead of routing wires. The pin coordinate is computed from the library
   pin offset with one transform: `schematic_xy = instance_xy + (lib_x, −lib_y)`
   (KiCad symbol libs are Y-up; the sheet is Y-down — see `pinpt`). Two pins that
   carry the same net just get the same label string. **Connectivity is by net
   name + coincident pins, not by drawn wire geometry.** This is the key trick:
   no wire routing to get wrong, and ERC proves it for you.

3. **Deterministic UUIDs.** Every element needs a UUID; the script derives them
   with `uuid5` over a fixed namespace and a fixed root-sheet UUID, so re-running
   reproduces the committed file byte-for-byte (no spurious diffs).

4. **Prove it with ERC.** ERC with `pin_not_connected=error` and
   `label_dangling=error` is what *catches a missed connection*: a pin you forgot
   to label dangles → ERC error. So a clean ERC run (0/0) is strong evidence the
   topology is wired as intended — that is why this method is trustworthy without
   eyeballing wires.

You do not have to use a generator. For small edits, Read the `.kicad_sch`, Edit
the S-expression in place, and re-run ERC. The generator just makes a large,
repetitive capture (3 identical channels) reproducible.

### Schematic loop, concretely
```bash
python hardware/gen_ref200_rtd_sch.py          # (re)author the .kicad_sch
bash scripts/erc.sh s005                        # validate -> reports/erc/erc_s005.json
bash scripts/netlist.sh                         # export netlist, then read it and
                                                # check nets node-by-node (Track E did this)
"/c/Program Files/KiCad/10.0/bin/kicad-cli.exe" \
  sch export svg --output reports/sch.svg hardware/ref200-rtd.kicad_sch   # render to eyeball
```
Read the JSON report and the netlist back. A passing ERC + a netlist whose nets
match [board_spec.md](board_spec.md) is your "it works."

> Caveat the generator itself states: once the GUI is used for real layout work,
> **the `.kicad_sch` is the source of truth** and re-running the generator
> overwrites GUI edits. Use the generator for the initial capture; after that,
> Edit the file in place.

## Layout / PCB — how to approach it

Same principle: `.kicad_pcb` is **S-expression text** (footprints, pads, tracks,
zones, board outline are all nodes). Three viable levels of effort:

1. **File authoring (works headless, like the schematic).** Start a `.kicad_pcb`
   from the netlist, place footprints by writing `(footprint ...)` blocks at
   `(at x y rot)` coordinates, draw the board outline on `Edge.Cuts`, then
   validate with `bash scripts/drc.sh`. Good for: footprint **placement**,
   board outline, mounting holes, ratsnest sanity. This is the analog of the
   schematic method and is the most reliable agent-only path.

2. **The `pcbnew` Python module (this *is* a real API — PCB only).** Unlike
   Eeschema, KiCad ships a SWIG `pcbnew` Python module usable from KiCad's
   bundled Python without the GUI running. It can load/create a board, place and
   move footprints, set net classes, add zones, and save. Use it when
   programmatic placement is cleaner than hand-writing S-expressions. (There is
   also the newer IPC API / `kicad-python` that talks to a *running* KiCad — less
   useful for a headless agent; prefer the `pcbnew` module or file authoring.)

3. **Hand off detailed routing + copper pours to the GUI / a human.** Be honest
   about the limit: laying out controlled-impedance traces, hand-routing dense
   nets, and tuning pours by writing raw S-expressions is slow and error-prone.
   Do the placement and constraints programmatically, then let routing happen in
   the GUI — and still gate the result with `scripts/drc.sh`.

### Layout-specific intent for THIS board (from the Track E handoff)
The PCB must honor (see [board_spec.md](board_spec.md) and the
[Track E log](sessions/trackE.md) "Next action"):
- **Star ground** — all Force− returns + LDO GND + T7 GND meet at one point.
- **R_ref tight to its REF200**, thermally quiet (it sets ratiometric accuracy).
- **Per-channel sense RC physically before the mux**; matched sense series-R pair.
- **4-wire Force/Sense routed as separate pairs** to each terminal `Jn`.

### Layout loop, concretely
```bash

"$KCLI" pcb export gerbers --output reports/fab/ hardware/ref200-rtd.kicad_pcb  # fab check
```

## What the headless loop *cannot* catch — compensate for it

`kicad-cli` checks **electrical/geometric rules**, not **design intent**. ERC
passes a wrong-but-legal schematic; DRC passes an ugly-but-legal layout. So:

- **Render and read it back.** Export SVG/PDF (`sch export svg`, `pcb export
  svg/pdf`, `pcb render`) and actually look, or attach it for Lucas to review.
- **Cross-check the netlist against the spec** node by node (Track E did exactly
  this — see its log). The netlist is the ground truth for "is it wired right."
- **Trace every choice to [board_spec.md](board_spec.md)**; record deviations in
  the session log. That is the project's hard rule, and it is your stand-in for
  engineering review.

## TL;DR for the skeptical session

- "No KiCad scripting API" is only true for **Eeschema**, and it doesn't block you.
- KiCad files are **text** → author/edit them directly (Write/Edit or a Python generator).
- `kicad-cli` is your **headless validator/exporter** → ERC, DRC, netlist, renders.
- Schematic trick: **net labels at exact pin coordinates**, deterministic UUIDs,
  prove connectivity with ERC. Working example: [gen_ref200_rtd_sch.py](../hardware/gen_ref200_rtd_sch.py).
- PCB: same file-as-text approach for placement/outline, **`pcbnew` Python module**
  exists for programmatic PCB work, hand off dense routing to the GUI, gate with DRC.
- Always run the gates: `bash scripts/run_gates`. Never commit ERC/DRC **errors**.
